# Step 6 — Design System Anti-Patterns

The bundle failure modes that produce slop or break downstream consumers.

| Pattern | Instead |
|---------|---------|
| Tokens-by-the-yard (30 colors, 12 type scales, 8 radii) | Cap at 8–14 colors, 5–7 type sizes, 2–4 radii in v1. Force the early hard choices; expand later when the prototype proves the need. |
| Visual token names (`--color-blue-500`, `--color-orange-warm`) | Semantic names (`--color-primary`, `--color-warning`, `--color-foreground-tertiary`). Raw primitives can co-exist (`--primitive-blue-500`) but the consuming surface reads semantic. A rebrand changes the primitive value, not the consuming code. |
| Hardcoded px values inside tokens (`--space-md: 16px`) | Use rem (`--space-md: 1rem`). Exception: `1px` for borders. Rem scales with user font-size preference; px doesn't. |
| Mixed brand identities in one design system | One brand per design system. Catalog path uses one anchor. Custom path builds a cohesive system. If the brand-book is genuinely hybrid, document the hybrid composition explicitly with the boundary rule (e.g. "marketing surface uses Spacious tokens; product surface uses Compact tokens"). |
| Tokens with no provenance | Every value traces back: catalog DESIGN.md citation in `## Catalog Lineage`, brand-book color story / visual direction line, OR a step-4 audit fix with finding ID. Tokens without provenance are inventions and rot quickly. |
| Skipping `## Audit Response` because no findings landed | Emit the explicit empty-state line (`*No design-system-routed findings from step 4 audit.*`). Section presence is enforced; an absent section signals "didn't read step 4 frontmatter", which is worse than "read it, nothing applied". |
| Ignoring step-4 frontmatter when it's present | When step 4 emits the YAML frontmatter (measurable mode), the `findings[]` tagged `fix_skill_hint: "design-system"` are NOT optional — they're the structured handoff that closes the audit→token-edit loop. Skipping them re-introduces the failure the deep port was designed to prevent. |
| `tokens.css` without intent comments | Each token gets a comment naming what it represents and (when applicable) the originating finding ID. `--color-foreground-tertiary: oklch(0.55 …);` is a value; `--color-foreground-tertiary: oklch(0.55 …); /* fix(F-07): brightened from 0.50 to lift surface contrast 3.89:1 → 5.10:1 */` is a system. |
| Components describing implementation (`React.forwardRef`, `className`) | Components describe **anatomy** + **states** + **tokens consumed**. The implementation layer is the consuming surface's choice; the spec is framework-agnostic. |
| Ship Storybook / Histoire / Ladle stories from this step | Those are downstream consumer-side artifacts. Step 6 produces the spec + tokens + anatomy; consumers build their own component implementations + stories. |
| Mandate a single rigid section format for `design-system.md` verbatim | This step's section floor is Overview / Tokens / Components / Patterns / Accessibility Floor / Audit Response, optionally Catalog Lineage. Product-class extensions live in `references/section-floor.md`. The multi-artifact bundle splits the content across `design-system.md` + `tokens.css` + `components.md` — don't cram everything into the spec file. |
| `components.md` as a flat list of component names | Per-component block with Anatomy + Variants + States + Tokens consumed. Without these, downstream steps (7 + 13) have to re-derive how to render each component. |
| Ship `tokens.json` when no consumer needs it | Optional fourth file. Skip when the consuming codebase doesn't need a machine-readable export (a project that just imports `tokens.css` directly doesn't need JSON). Only generate when Style Dictionary, Tailwind config generation, or Figma variables are real consumers. |
| Catalog path without citing the anchor | When path is `catalog`, `## Catalog Lineage` MUST name the anchor system(s), what was taken verbatim, what was deviated. Anchorless catalog use is plagiarism + makes deviations invisible. |
| Inventing components the prototype doesn't use | Read step-2 hi-fi screens for the actual component inventory. Adding a Tab component because "every design system has tabs" produces 4 KB of dead spec. |
| Density picked by aesthetic preference | Density is product-class-driven (compact for data-dense / dev tools; comfortable for B2B SaaS; spacious for marketing / luxury). When the product is hybrid, ship two scales with documented usage rules. |
| Accessibility floor as "we target WCAG AA" without testing | Compute contrast for every token pair the system uses. Document failures + fixes in `## Audit Response` (or the design-system Accessibility Floor section if no step-4 audit ran). "Target" without "tested" is a wish. |
